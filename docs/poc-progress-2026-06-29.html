§24 个采集桶 · 每个怎么干的

每个博客先分进对应的"桶"(基于域名 + probe 探测结果)· 各自走不同 handler · 抽不同字段的逻辑。详细看下面 4 个 box。

桶 1 · 📝 Medium 平台 · 171 源(165 unique RSS)

策略:RSS endpoint + 1-to-N mapping Crawler:mediumCrawler 本次抓到:292 条 handler:src/handlers/medium.ts

怎么抓:

1. URL 转换 · medium.com/@user → medium.com/feed/@user(RSS endpoint)
2. 用 Crawlee CheerioCrawler + ImpitHttpClient 抓 XML
3. cheerio 解 XML 抽 <item> 列表(每个 item 一个 post)
4. 1-to-N mapping · 多个 token 共一个 Medium publication → 一次抓 RSS 输出 N 条 dataset(每个 token 一份)

抽取字段:

字段取自例
标题<item><title>"SuperEx Web3 Wallet User Guide(XIV)"
URL<item><link>https://superex.medium.com/...
发布时间<item><pubDate>(RFC 822)"Mon, 29 Jun 2026 07:36:39 GMT"
作者<item><dc:creator>"SuperEx"
正文摘要<description> 或 <content:encoded> 前 280 字 · 剥 HTML"The Road to TGE: Yooldo's Path..."
分类多个 <category>["account-abstraction", "ethereum"]
GUID<guid>medium 文章唯一 ID · 后续去重用

真实样本:

最新 · token_id=12718 · base_symbol=UVXY
SuperEx Guide: SuperEx Web3 Wallet User Guide(XIV)
发布:Mon, 29 Jun 2026 07:36:39 GMT(今天)
作者:SuperEx · URL:https://superex.medium.com/superex-guide-...-c8ae9663cf66?source=rss-...

这次的成功率(why 只 292 条): Medium 实测对 RSS endpoint 也限速(IP 维度)· 我们这次提速后(60 RPM + 3 并发 · sessionPool 关闭)触发了 Medium IP-based limit · 大部分 RSS 被 403。代理池来后用多 IP 轮换可解。

桶 2 · 🗺️ sitemap 自部署站 · 294 源 · 2,485 个 URL

策略:读站点 sitemap.xml 拿 article URL 列表 Crawler:generalCrawler 本次抓到:2,349 条(占总 86%) handler:src/handlers/default.ts

怎么抓:

1. probe 阶段已经为每个源探到 sitemap_url(/sitemap.xml / /sitemap_index.xml / /sitemap.txt)
2. 并发 Sitemap.load() 拉所有 sitemap · 解析 XML 拿 URL 列表
3. 每源取 前 10 个 URL(可调 · 默认 10 · 防止某些站 sitemap 几万 URL 撑爆)
4. 入 generalCrawler 队列 · 每个 URL 抓 article 页 · 走 default handler 抽 og

抽取字段(走 default handler · 同桶 4):

字段取自
标题og:title → <title> 兜底
描述og:description → meta[name=description]
图片og:image
站点名og:site_name
类型og:type(article / website / ...)
发布时间article:published_time(很多 sitemap 列的不是 article · 没这字段)

真实样本:

token_id=12970 · base_symbol=SONY · 来自 dinari.com sitemap
Dinari Inc. Privacy Policy
URL:https://dinari.com/privacy
⚠️ 这是 sitemap 列了非 article 页面带来的"杂项" · 后续可加 URL pattern 过滤(只要 /blog/ /news/ 路径)

桶 3 · 🔍 heuristic 启发式 · 87 源(无 og)

策略:多重 fallback + RSS auto-discovery Crawler:generalCrawler(label='heuristic') 本次抓到:26 条 · 96% 有 title handler:src/handlers/heuristic.ts

抽取字段(多重 fallback · 谁有用谁):

字段取自(优先级从高到低)
标题og:title → meta[name=title] → <title> → <h1>
描述og:description → meta[name=description] → <article p>首段 → <main p>首段 → <p>首段(各取 280 字)
图片og:image → twitter:image → <article img> → <main img> → 各种 <link rel=icon> → 首张 <img>
发布时间article:published_time → meta[itemprop=datePublished] → <time datetime> → meta[name=date] → meta[name=publish_date] → meta[name=pubdate]
RSS 自动发现<link rel="alternate" type="application/rss+xml"> + atom · 找到就记录 · 后续可走 RSS 路径
候选 article 链接站内 <article a> <main a> .post a .entry a 前 5 个 · 后续可深爬

真实样本:

token_id=12158 · base_symbol=POWER_2
Power Protocol · description: Power Up Your Play.
image:https://powerprotocol.xyz/_next/image?url=...power_protocol_logo-long.webp
sample_article_links:3 个外部交易所链接(不是 article · 需后续优化)

验证结论:思路 OK · 96% 有 title · 但 RSS auto-discovery 0% 命中(这批 og=none 站普遍不暴露 RSS link)· 87 源里 61 个被反爬挡了 · 代理池可救。

6.2 代理池后的工程改动(我后续做)

  1. 加 ProxyConfiguration:mediumCrawler 和 generalCrawler 各接一份代理池 · Crawlee 自带支持
  2. Registry 加 interval_ms 字段:按 host_platform / 发文频率自动分配(Medium 类 = 5 min · 自部署 = 30 min · 静态 = 2h)
  3. systemd timer 改成 per-source 调度:不再统一 3h 全量跑 · 而是每分钟检查"哪些源到时间了" · 增量抓
  4. 反爬 fallback:同一源连续失败 N 次 → 自动降级走 Playwright(P3.5)
  5. 监控:每 IP 的命中率 / 失败率 / cooldown · dashboard 显示
